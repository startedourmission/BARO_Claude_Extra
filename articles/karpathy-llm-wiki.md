# 카파시의 LLM 위키 만들기

> RAG는 매번 다시 찾고, 위키는 한 번 정리하면 쌓인다. 안드레 카파시가 제안한 "LLM이 스스로 관리하는 개인 위키" 패턴을 따라 해봅니다.

## 들어가며

대부분의 사람들이 LLM으로 문서를 다루는 방식은 RAG(Retrieval-Augmented Generation)입니다. 파일을 업로드하고, LLM이 질문 시점마다 관련 청크를 검색해 답을 생성하죠. 잘 동작하지만 중요한 한계가 있습니다. **매 질문마다 LLM이 지식을 처음부터 다시 조립한다**는 점입니다. 다섯 개 문서를 종합해야 하는 미묘한 질문을 하면, LLM은 매번 그 다섯 조각을 다시 찾아 붙입니다. 축적이 없습니다.

[안드레 카파시](https://karpathy.ai/)가 [공개한 아이디어](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f)는 이 구도를 뒤집습니다. LLM이 질의 시점에 지식을 조립하는 대신, **지식을 한 번 정리해 마크다운 위키로 축적해 두고, 이후의 질의는 이미 정리된 위키를 읽게 하자**는 겁니다.

이 글에서는 그 패턴을 클로드 코드로 어떻게 구현하는지 실전 예제와 함께 정리합니다.

## 왜 RAG를 넘어서야 하는가

RAG의 문제를 조금 더 구체적으로 짚어봅니다.

- **누적되지 않는다** — 어제 한 질문의 분석 결과가 오늘 사라집니다. 같은 문서 묶음을 반복 질의해도 매번 처음부터입니다.
- **모순이 남는다** — 여러 문서 간 충돌이 있어도 검색 단계에서는 감지되지 않습니다. 답변이 왔다 갔다 할 뿐입니다.
- **연결이 사라진다** — 문서 A의 개념과 문서 B의 개념이 사실 같은 것을 가리킨다는 걸 발견해도, 다음 질의에서는 그 연결이 다시 유실됩니다.
- **벡터 DB의 오버헤드** — 임베딩, 인덱싱, 재인덱싱, 벡터스토어 운영의 복잡성을 유지해야 합니다.

카파시는 "중간 규모(~100개 소스, 수백 쪽 수준)에서는 놀라울 정도로 잘 동작한다"고 말합니다. 벡터 검색 없이 **인덱스 파일 기반 탐색**만으로도 충분하다는 것이죠.

## 3계층 구조

카파시의 위키는 세 층으로 이루어집니다.

### 1. Raw sources (원본)

손대지 않는 원천 자료입니다. 논문 PDF, 기사, 책, 데이터 파일 등을 `sources/` 폴더에 그대로 둡니다. 버전이 바뀌거나 갱신되는 일이 있어도, 이 폴더의 파일은 "박제된" 증거로 남습니다.

### 2. The wiki (위키)

LLM이 생성·유지하는 마크다운 파일들입니다. 소스에서 추출한 요약, 핵심 엔티티, 개념, 그리고 서로를 연결하는 링크가 여기에 쌓입니다.

```
wiki/
├── index.md              # 전체 목차
├── concepts/             # 개념별 페이지
│   ├── attention.md
│   ├── transformer.md
│   └── tokenization.md
├── entities/             # 인물·조직·제품
│   ├── karpathy.md
│   └── openai.md
└── summaries/            # 개별 소스 요약
    ├── attention-is-all-you-need.md
    └── gpt-3-paper.md
```

각 페이지는 위키링크(`[[attention]]`)로 서로를 가리켜 그래프를 이룹니다.

### 3. The schema (CLAUDE.md)

위키를 어떻게 관리할지 정의한 메타 문서입니다. LLM이 **매번 같은 규칙으로** 위키를 확장·유지하도록 보장합니다. 폴더 구조, 페이지 템플릿, 작명 규칙, 세 가지 연산의 구체적 절차가 여기에 들어갑니다.

## 3가지 연산: Ingest / Query / Lint

### Ingest — 새 소스를 읽고 위키를 갱신한다

```text
sources/에 새로 들어온 PDF를 읽고, 위키에 아래 순서로 반영해 줘:

1. summaries/<slug>.md 생성 — 핵심 주장, 방법, 결과, 한계
2. 언급된 개념·인물은 concepts/, entities/에 페이지가 있으면 업데이트,
   없으면 새로 생성
3. 기존 페이지와의 교차 링크를 [[wiki-link]]로 추가
4. index.md에 새 페이지들을 등록
5. logs/ingest-YYYY-MM-DD.md에 오늘 반영한 내용을 요약
```

중요한 건 **한 번 정리한 지식은 다시 정리하지 않는다**는 점입니다. 다음 소스가 들어와도 LLM은 관련 페이지만 **증분 갱신**합니다.

### Query — 위키를 뒤져 답한다

```text
"Transformer의 positional encoding이 왜 sinusoidal인지"
wiki/에서 관련 페이지들을 읽고 답해 줘.
답변에 참고한 페이지를 citation으로 달아.
새로운 통찰이 있으면 concepts/positional-encoding.md에
"Notes from 2026-04-17 query"로 추가해.
```

질의 자체가 **위키를 성장시키는 계기**가 됩니다. 시간이 갈수록 위키는 더 풍부해지고, 같은 질문에 대한 답변 품질도 올라갑니다.

### Lint — 건강 검진

```text
wiki/ 전체를 점검해 줘:

- 서로 모순되는 주장 (같은 사실에 대해 다른 값을 쓰는 페이지)
- 오래된 정보 (updated-at이 6개월 이상 지난 페이지 중
  최근 소스와 충돌 가능성이 있는 것)
- 링크가 끊긴 페이지 (아무도 가리키지 않는 orphan)
- 생성이 누락된 교차 링크 (본문에서 개념을 언급했지만 [[link]]가 안 걸린 것)

점검 결과를 lint-report.md로 저장해.
```

위키가 자라면서 생기는 엔트로피를 **주기적으로 LLM에게 청소시키는** 것이 핵심입니다. 사람이 직접 관리하면 부담되지만, LLM은 이런 반복적 정리를 지치지 않고 수행합니다.

## 실전: CLAUDE.md 작성하기

위키 폴더 루트에 아래와 같은 `CLAUDE.md`를 두면, 이후 클로드 코드 세션은 자동으로 이 규칙을 따릅니다.

```markdown
# LLM Wiki Schema

이 저장소는 개인 지식 위키입니다.
세션을 시작하면 아래 구조와 규칙을 따라주세요.

## 폴더 구조

- `sources/` — 원본 자료. 절대 수정하지 않음
- `wiki/concepts/` — 추상 개념 페이지
- `wiki/entities/` — 인물·조직·제품 페이지
- `wiki/summaries/` — 개별 소스 요약
- `wiki/index.md` — 전체 목차
- `logs/` — 날짜별 작업 기록

## 페이지 템플릿

모든 위키 페이지는 아래 프론트매터로 시작:

\`\`\`yaml
---
title: 페이지 제목
type: concept | entity | summary
created: YYYY-MM-DD
updated: YYYY-MM-DD
sources: [사용한 원본 파일 목록]
---
\`\`\`

## 작명 규칙

- 파일명: kebab-case (`positional-encoding.md`)
- 위키링크: `[[positional-encoding]]` 형태
- 제목: 한글 우선, 영문 병기는 괄호로

## 연산 정의

### /ingest <파일경로>
sources/의 파일을 읽고 summaries/에 요약 페이지 생성.
concepts·entities 페이지 증분 갱신.
index.md와 logs/ingest-{오늘날짜}.md 업데이트.

### /query <질문>
wiki/를 탐색해 답변 생성. 인용 페이지 표기.
새 통찰은 관련 페이지 하단 "Notes from {날짜}" 섹션에 추가.

### /lint
모순, 고아 페이지, 끊긴 링크, 누락된 교차 링크 점검.
결과를 lint-report-{오늘날짜}.md로 저장.

## 금지 사항

- 원본(sources/) 수정 금지
- 벡터 DB나 임베딩 인프라 도입 금지 (인덱스 파일로 충분)
- 페이지를 다시 처음부터 쓰지 말 것. 항상 증분 갱신.
```

이 한 장의 문서가 곧 **위키 운영 규약**입니다. 팀으로 운영한다면 git에 커밋해 팀원 모두가 같은 규칙으로 위키를 확장하도록 만들 수 있습니다.

## Obsidian과 결합하기

카파시 자신이 쓰는 뷰어는 [Obsidian](https://obsidian.md/)입니다. Obsidian은 폴더 하나를 "vault"로 지정해 마크다운 파일을 트리 형태로 보여주는 무료 앱입니다. 세 가지 기능이 이 패턴과 잘 맞습니다.

1. **Graph View** — 페이지 간 `[[링크]]`를 노드·엣지로 시각화. 위키의 구조와 빈 영역을 한눈에 봅니다.
2. **Backlinks** — 이 페이지를 가리키는 다른 페이지 목록 자동 표시. 위키 탐색의 기본 동선이 됩니다.
3. **Tag & Search** — 프론트매터의 `type`, `tag`로 필터링. LLM이 점검(lint) 쿼리를 돌릴 때도 같은 필터를 그대로 씁니다.

클로드 코드는 파일을 쓰고, Obsidian은 읽기/탐색을 담당하는 구도입니다. 두 도구가 **같은 폴더를 공유**하므로 충돌이 없습니다.

## 쓸모 있는 활용 시나리오

- **책 읽으며 정리하기** — 챕터 하나를 읽을 때마다 `/ingest`. 완독 후엔 책 한 권 분량의 개인 위키가 남습니다.
- **논문 리서치** — 관련 논문 20편을 `sources/`에 넣고 `/ingest` 반복. 개념·방법론 페이지가 교차 참조되며 자라납니다.
- **개인 일지** — 일기·회고·메모를 마크다운으로 기록하고, LLM이 주기적으로 주제별 페이지를 합성해 줍니다.
- **기술 문서 학습** — 새 프레임워크 문서를 통째로 넣고, 개념 위키를 만들게 한 뒤 나중에 코드 작성 시 이 위키를 CLAUDE.md에서 참조합니다.
- **팀 지식 베이스** — 회의록·결정 문서·장애 보고서를 소스로 넣고, 정기적 `/lint`로 변경 이력과 모순을 관리합니다.

## RAG와 무엇이 다른가

| 항목 | RAG | LLM Wiki |
|------|-----|----------|
| 지식 저장 | 벡터 DB의 청크 | 마크다운 파일 |
| 질의 시 동작 | 청크 검색 + 생성 | 위키 페이지 읽기 |
| 축적 | 매번 새로 조립 | 한 번 정리, 계속 누적 |
| 교차 참조 | 자동 유지 어려움 | 위키링크로 명시 |
| 모순 감지 | 사실상 불가 | lint 연산으로 주기 점검 |
| 인프라 | 임베딩·벡터스토어 | 파일시스템 + 에이전트 |
| 투명성 | 블랙박스 | 사람도 직접 읽고 수정 가능 |

카파시는 "~100 소스, 수백 쪽" 규모를 잘 되는 지점으로 꼽습니다. 그보다 커지면 인덱스 전략을 고민해야 하지만, 대부분의 개인·소규모 팀에게는 이 정도면 차고 넘칩니다.

## 시작해보기

1. 빈 폴더를 만들고 `sources/`, `wiki/`, `logs/`를 만듭니다.
2. 루트에 위의 `CLAUDE.md` 예시를 복사해 둡니다.
3. `sources/`에 정리하고 싶은 자료 2~3개를 넣습니다.
4. 해당 폴더에서 `claude`를 실행해 `/ingest sources/첫번째파일.pdf`를 요청합니다.
5. 몇 개 넣은 뒤 `/query`로 질문해 보고, 주기적으로 `/lint`를 돌립니다.
6. (선택) Obsidian을 설치해 같은 폴더를 vault로 지정합니다.

며칠만 써봐도 "내가 읽은 것들이 내 바깥에 쌓인다"는 감각을 느낄 수 있습니다. 그게 이 패턴의 핵심입니다.

## 마치며

LLM Wiki는 "LLM을 검색 엔진처럼 쓰지 말고, **편집자**처럼 쓰자"는 제안입니다. 지식을 매번 조립하게 하지 말고, **한 번 정리해 보관해 두라고 시키자**는 것이죠. 벡터 DB의 복잡성도, RAG의 재계산 비용도 필요 없습니다. 파일시스템과 마크다운, 그리고 꾸준히 규칙을 따르는 에이전트만 있으면 됩니다.

카파시가 강조한 말 한 줄로 마무리합니다. "사람은 소스를 고르고 질문을 하고, 나머지 지루한 일은 LLM이 한다."

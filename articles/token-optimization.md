# 클로드 토큰 절약 완전 가이드

> 같은 작업을 **10배 싸게** 끝내는 법. 캐싱, 컨텍스트 위생, 서브에이전트, 스크립트 이관, 프롬프트 설계까지 12가지 전략을 하나도 빠짐없이 정리합니다.

## 들어가며

클로드 코드를 며칠 써 보면 체감합니다. **토큰은 잘못 쓰면 정말 빠르게 탑니다.** 하루 평범하게 썼는데 $30 나오는 날이 있고, 같은 양 작업했는데 $3로 끝나는 날도 있습니다. 차이는 기량이 아니라 **컨텍스트 관리 습관**입니다.

이 글은 실무에서 검증된 토큰 절약 기법 12가지를 정리합니다. 개별 기법마다 효과 크기가 다르고, 같이 썼을 때 곱으로 작용하는 것도 있습니다. 전부 적용하면 **평균 토큰 사용량 50~80% 감소**가 보수적인 기대치입니다.

## 토큰이 어디서 타는가 — 다섯 가지 누수

먼저 상대를 알아야 합니다. 토큰이 낭비되는 주요 지점:

1. **반복 전송되는 고정 컨텍스트** — CLAUDE.md, 도구 스키마, 시스템 프롬프트가 매 턴마다 통째로 전송
2. **오래된 대화의 잔여** — 3시간 전 해결한 문제의 로그·에러·diff가 계속 컨텍스트에 달려 다님
3. **과도한 모델 선택** — Haiku로 끝날 일에 Opus를 씀
4. **지나친 사고** — 단순 작업에 `ultrathink` 붙임
5. **방만한 프롬프트** — "적절히 해 줘"가 LLM을 헤매게 해 토큰을 더 씀

이 다섯 가지를 각각 대응하는 게 아래 12 전략입니다.

## 전략 1 — 프롬프트 캐싱 (최대 효과)

**Anthropic의 가격 기능 중 가장 임팩트 큼.** 고정된 시스템 프롬프트·문서·도구 정의를 5분간 캐시해, 다음 요청이 캐시를 읽으면 **입력 토큰 비용의 10%**만 냅니다. **90% 절감**이 표준.

### 클로드 코드에서의 자동 캐싱

클로드 코드는 개발자가 별도 설정 안 해도 아래를 자동으로 캐시합니다.

- 시스템 프롬프트
- CLAUDE.md 계층 전체
- 활성화된 스킬의 본문
- MCP 도구 스키마
- 최근 N턴의 대화 히스토리

### 캐시 히트율 최대화 팁

- **쉬지 않고 이어서 작업** — 캐시 TTL 5분. 점심 먹고 오면 날아가 있음
- **CLAUDE.md 안정적으로 유지** — 자주 바꾸면 캐시 무효화 빈발
- **`/clear` 신중하게** — 이후 첫 요청은 무조건 캐시 미스
- **긴 세션 도중 `/compact`** — 캐시가 warm한 상태에서 압축해야 이후 요청도 warm

### 확인 방법

```
/cost
```

출력에 `cache read tokens` / `cache write tokens` 비율이 나옵니다. **캐시 읽기 비율이 70%+면 건강한 상태.** 50% 아래면 점검 필요.

### 2026년 3월 캐싱 인시던트

2026년 3월 Anthropic 측 버그 두 건으로 캐시 계산이 10~20배 인플레이션된 사건이 있었습니다. 현재는 해결됐지만, **`/cost`가 평소보다 이상하게 높으면** 단순히 내 습관 문제가 아닐 수 있으니 공식 상태 페이지를 확인하세요.

## 전략 2 — 모델을 일에 맞춰 고르기

아키텍처는 Opus, 리팩터는 Sonnet, 분류·요약·renamepublic은 Haiku. **세 모델 가격차가 5~15배**라 작업 성격과 안 맞는 모델은 직격입니다.

### 실측 기준 권장

| 작업 | 권장 | 왜 |
|------|------|-----|
| 새 기능 설계 | Opus | 깊은 추론 |
| 일반 구현 | Sonnet | 스윗스팟 |
| 버그 수정 (소규모) | Sonnet | 충분 |
| 파일 일괄 리네임 | Haiku | 단순 반복 |
| 커밋 메시지 작성 | Haiku | 짧고 규격적 |
| 대량 문서 분류 | Haiku | 배치에 강함 |
| 복잡 디버깅 | Opus | 장문 추론 필요 |

### 동적 전환

세션 중 모델 바꾸기:

```
/model haiku      # 빠른 작업 전
/model sonnet     # 일반 복귀
/model opus       # 복잡 전환
```

작업 끝에 Haiku로 되돌려 두는 습관이 장기 비용에 큰 차이.

### 스킬별 고정

자주 쓰는 스킬은 프론트매터에 모델 박아 두세요.

```yaml
---
name: quick-rename
model: haiku
---
```

```yaml
---
name: architecture-review
model: opus
effort: high
---
```

## 전략 3 — 사고 토큰 상한

`think`·`ultrathink` 같은 Extended Thinking은 토큰을 크게 씁니다. 기본 대비 `ultrathink`는 10~20배까지 사고 토큰 소비 가능.

### 환경변수로 상한

```bash
export MAX_THINKING_TOKENS=8000
```

이보다 많이 사고해도 이 상한에서 잘립니다. 데일리 세션에는 8000~10000이 합리적.

### `/effort` 커맨드

세션 단위로 노력 레벨 조정:

```
/effort low       # 빠르고 싸게
/effort medium    # 기본
/effort high      # 어려운 문제
/effort xhigh     # 드물게
/effort max       # 정말 예외
```

### 사고 모드 쓰는 규율

| 상황 | 권장 |
|------|------|
| 단순 코드 생성 | 없음 |
| 번역·요약 | 없음 |
| 일상 버그 | `think` |
| 복잡 버그 | `think hard` |
| 아키텍처 결정 | `ultrathink` |

**매 프롬프트에 `ultrathink` 붙이는 습관이 가장 큰 낭비.** 판단이 필요한 지점에만.

## 전략 4 — 컨텍스트 위생

### `/clear` vs `/compact`

- **`/clear`** — 완전 초기화. 새 주제 시작할 때
- **`/compact`** — 요약 압축. 연관 작업 이어갈 때

주제가 바뀌면 망설이지 말고 `/clear`. 같은 주제를 길게 이어가면 도중에 `/compact`.

### `/rename` + `/resume` 콤보

```
# 현재 작업 잠시 보류하고 싶을 때
/rename "auth-refactor-2026-04-19"
/clear          # 컨텍스트 비움
# 새 작업 진행
...
/resume         # 목록에서 auth-refactor 재개
```

세션 전환을 깔끔하게 하면서도 돌아올 길을 남김.

### `/compact` 지시 커스터마이즈

그냥 `/compact`가 아니라 **무엇을 보존할지** 지정:

```
/compact 코드 샘플과 API 시그니처는 유지, 대화체 탐색은 버려.
```

중요 부분이 요약 과정에서 희석되는 것을 방지합니다.

### 수동 압축이 자동보다 낫다

자동 압축은 컨텍스트가 꽉 차서야 동작합니다. 이미 비싼 호출을 한 다음이죠. **작업 자연 단위**마다 수동으로 `/compact`하는 편이 품질·비용 모두 유리.

### CLAUDE.md 슬림화

루트 CLAUDE.md가 500줄이면 **매 턴마다** 그 500줄이 캐시에서 읽힙니다(캐시 히트라도 10% 비용). 불필요한 섹션은 빼고, 상세는 하위 디렉터리 CLAUDE.md나 `@import`된 문서로 이동.

**목표: 루트 CLAUDE.md는 150줄 이내.**

## 전략 5 — 서브에이전트로 컨텍스트 격리

거대한 탐색·조사 작업은 **메인에서 직접 하지 말고** 서브에이전트에 위임하세요. 서브는 독립 컨텍스트에서 파일을 읽고 결과 **요약만** 메인으로 돌려줍니다.

### 전형적 절감 사례

메인에서 직접:

```
"src/ 전체 뒤져서 auth 관련 파일 찾아 줘"
→ 40개 파일이 메인 컨텍스트에 로드 (15,000 토큰)
→ 이후 세션 내내 그 15,000 토큰이 반복 전송
```

서브에이전트 위임:

```
Agent(Explore, "src/ 에서 auth 관련 파일 찾아 요약")
→ 서브에이전트 컨텍스트에 40개 로드 (격리됨)
→ 메인은 요약 500 토큰만 받음
→ 이후 반복 비용이 30배 차이
```

### 싼 모델로 서브 돌리기

단순 탐색·문서 조회·로그 파싱 같은 서브는 Haiku로 충분:

```yaml
---
name: log-summarizer
model: haiku
tools: Read, Grep
---
```

메인은 Opus 유지하되 서브는 Haiku. **메인 1 Opus + 서브 5 Haiku 병렬** 조합이 품질·비용 스윗스팟.

### 병렬 실행 효과

5개 서브를 동시에 띄우면 병렬 실행이라 **전체 시간은 1개분**. 하지만 토큰 소비는 5개 각각 집계되므로 순차·병렬 토큰 차이는 없음. 시간과 토큰은 다른 축.

## 전략 6 — 결정론 부분을 스크립트로

**스크립트는 컨텍스트에 로드되지 않습니다.** LLM은 Bash로 실행만 하고 그 출력만 토큰으로 받습니다.

### 예시 — 로그 분석

**나쁜 방식:**
- SKILL.md에 정규식·파싱 규칙을 자연어로 풀어씀
- LLM이 매번 `Read`로 로그 로드 → 한 줄씩 판정
- 100MB 로그면 아예 불가능

**좋은 방식:**
- `scripts/analyze.py`에 파이썬으로 파싱 구현
- SKILL.md엔 "분석 필요 시 `scripts/analyze.py` 실행"만
- LLM은 결과 **요약만** 받음 (수백 바이트)

**토큰 10~100배 절감** 흔한 결과.

### uv + PEP 723로 자립 스크립트

```python
#!/usr/bin/env -S uv run --script
# /// script
# dependencies = ["pandas>=2"]
# ///

import pandas as pd
# 결정론 파싱 로직
```

상세는 활용 카테고리의 "스크립트로 에이전트 스킬 고도화하기" 참고.

## 전략 7 — MCP 도구 가지치기

MCP 서버 하나가 30~50개 도구를 노출하는 게 흔합니다. 다 켜 놓으면 **모든 도구 스키마가 매 턴마다 컨텍스트로** — 큰 누수.

### Profile 활용

`notebooklm-mcp` 같은 일부 서버는 프로파일 옵션 제공:

```bash
npx notebooklm-mcp config set profile minimal  # 5개 도구만
```

### 서버별로 꼭 필요한 도구만

프로젝트 `.mcp.json`에서:

```json
{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": { "ALLOWED_TOOLS": "create_issue,create_pull_request,get_issue" }
    }
  }
}
```

### 세션별 필요한 MCP만

개발 세션에 Slack MCP 필요 없다면 연결 끊기:

```
/mcp disconnect slack
```

다시 필요할 때 connect.

## 전략 8 — 프롬프트 구체화

**"적절히"**, **"잘"**, **"괜찮게"** — 이런 모호한 지시는 LLM이 판단에 토큰을 쓰게 만듭니다. 구체 지시가 짧고 싸고 정확.

### 나쁜 프롬프트

```
이 코드 개선해 줘
```

LLM이 "무엇을 개선? 성능? 가독성? 보안?" 고민하며 토큰 소비.

### 좋은 프롬프트

```
src/auth/login.ts에서 try-catch를 early return으로 리팩터.
에러 메시지 한국어로 고정.
테스트 건드리지 말 것.
```

- 대상 **파일 명시**
- 변경 **구체적 기법** 지정
- 비변경 **영역 제한**

### 템플릿 만들기

자주 쓰는 요청을 스킬로 추출:

```markdown
---
name: refactor
description: 특정 파일을 지정 패턴으로 리팩터
---

$0 파일에서 $1 패턴을 $2로 변환.
테스트 파일은 건드리지 말 것.
변경 후 `pnpm lint` 돌려 검증.
```

`/refactor src/auth/login.ts try-catch early-return` 한 줄로 호출.

## 전략 9 — 출력을 파일로 밀어내기

클로드가 생성하는 긴 결과물(리포트, 분석, 코드 대량 생성)을 대화에 그대로 남기면 **이후 모든 턴의 입력에 그게 포함**됩니다.

### 나쁜 패턴

```
"src/ 전체 분석 리포트 써 줘"
→ 클로드가 3000줄 리포트 대화에 출력
→ 그 뒤 모든 요청이 3000줄 포함해서 전송
```

### 좋은 패턴

```
"src/ 분석 리포트를 reports/analysis-2026-04-19.md에 저장해 줘.
대화에선 저장 완료 확인만 한 줄."
→ 대화엔 "reports/... 저장 완료" 한 줄만
→ 이후 토큰 부담 없음
→ 필요할 때만 그 파일을 @ 참조
```

긴 산출물은 **항상 파일로**.

## 전략 10 — 심볼 레벨 편집

파일 전체를 다시 쓰는 대신 **특정 부분만** 교체. 클로드 코드의 Edit 도구는 `old_string/new_string` 패턴으로 정확한 구간만 수정합니다.

### 큰 파일의 한 함수만 수정

나쁨: `Write`로 파일 전체 재작성 (파일 크기만큼 토큰)
좋음: `Edit`로 해당 함수 블록만 교체 (변경분만 토큰)

클로드는 기본적으로 Edit를 선호하지만, 때로 Write로 전체를 쓰려 할 때가 있습니다. CLAUDE.md에 명시:

```markdown
## 편집 원칙

- 파일 전체 재작성은 마지막 수단
- 변경 부분만 Edit로 교체
- 한 파일에 여러 Edit 호출도 무방
```

## 전략 11 — Batch API (비동기 대량 처리)

실시간이 아니어도 되는 작업은 **Batch API**로 50% 저렴하게:

- 문서 1000개 분류
- 코드베이스 전체 주석 생성
- 이메일 대량 요약

24시간 내 처리 보장, 가격은 동기 호출의 절반.

```python
import anthropic

client = anthropic.Anthropic()

batch = client.messages.batches.create(
    requests=[
        {"custom_id": f"doc-{i}",
         "params": {"model": "claude-haiku-4-5",
                    "max_tokens": 200,
                    "messages": [{"role": "user", "content": doc}]}}
        for i, doc in enumerate(documents)
    ]
)

# 나중에 결과 수거
results = client.messages.batches.results(batch.id)
```

Haiku + Batch 조합이면 동기 Sonnet 대비 **~15배 싸집니다.**

## 전략 12 — 설정 파일 슬림 유지

### 안 쓰는 스킬 정리

`~/.claude/skills/`에 쌓인 스킬 중 안 쓰는 건 이름만이라도 컨텍스트에 올라갑니다. 주기적으로 청소.

### 안 쓰는 플러그인 비활성화

```
/plugins
```

메뉴에서 불필요한 플러그인 비활성화.

### MCP 서버 최소화

개발 세션에 Slack·Google Calendar 같은 서비스 MCP가 꺼져 있으면 스키마가 컨텍스트에서 빠짐.

### CLAUDE.md @import 깊이 제한

```
CLAUDE.md → @docs/stack.md → @docs/deps.md → @package.json
```

세 층 타고 들어가면 기저엔 package.json 전체가 매 턴 전송. **중간 문서를 "필요 시 Read" 지시로 대체**하고 CLAUDE.md는 핵심만.

## 측정 — 습관으로 만들기

### 하루 리듬

- **세션 시작** — `/cost`로 0부터 시작 확인
- **작업 사이사이** — `/cost`로 누적 체크
- **작업 단위 종료** — `/compact` 또는 `/clear`

### 주간 리뷰

일주일에 한 번:

1. 세션 JSONL 파일 분석 (`~/.claude/projects/<프로젝트명>/`)
2. 토큰 많이 쓴 세션 상위 5개 열어 원인 파악
3. 반복 패턴 발견 시 스킬화 or CLAUDE.md 반영

### `/cost` 해석

```
Total input tokens: 150,000 (cache read: 120,000, cache write: 15,000)
Total output tokens: 8,000
Cache read ratio: 80%
Estimated cost: $0.45
```

- **Cache read ratio 70%+** → 건강
- **50% 아래** → 캐시 미스 빈발, 원인 찾기
- **Output 대비 input이 50배+** → 컨텍스트 비대, 정리 시점

## 효과 측정 — 전략 조합 가이드

우선순위(효과 큼 → 작음):

1. **전략 2 + 3 (모델·사고)** — 20~50% 즉시 절감
2. **전략 1 (캐싱 최적화)** — 캐시 히트율만 올려도 30~90% 추가 절감
3. **전략 4 (컨텍스트 위생)** — 길어지는 세션에서 큰 차이
4. **전략 5 (서브에이전트)** — 탐색 많은 프로젝트에서 30~80% 절감
5. **전략 6 (스크립트 이관)** — 반복 작업 있는 스킬에 10배 효과
6. 전략 7~12 — 각 1~15% 추가 최적화

**처음부터 전부 적용하려 하지 마세요.** 가장 큰 누수 찾아 한둘씩.

## 자주 하는 함정

### 캐싱 오해

5분 TTL이라 점심 먹고 오면 캐시 날아감. 복귀 첫 요청은 무조건 미스. 여기서 Opus로 큰 작업을 시작하면 비쌉니다.

### /compact 과다 사용

작업 흐름 한가운데 `/compact`하면 방금 읽은 파일 내용이 요약되어 LLM이 "뭐였더라" 다시 읽게 만듭니다. 작업 자연 단위 사이에 쓰세요.

### "싼 모델은 못 믿는다" 선입견

Haiku가 못하는 일만 Sonnet으로. Sonnet이 못하는 일만 Opus로. 대부분 작업에 Haiku가 충분합니다.

### 사고 토큰 무한 허용

`MAX_THINKING_TOKENS` 설정 안 하면 복잡 프롬프트에서 `ultrathink`가 수만 토큰을 태울 수 있습니다.

### 전체 파일 반복 Read

같은 세션에서 같은 파일을 여러 번 읽으면 그때마다 토큰. CLAUDE.md에 "한 번 읽은 파일은 다시 읽지 말 것" 규칙을 넣거나, 파일 내용을 변수처럼 요약해 두세요.

### 토큰 적게 쓰려 품질 포기

절약이 목표지 **학대가 목표가 아님**. Haiku로 돌렸다가 결과가 엉망이면 다시 Sonnet으로 돌려야 하고, 결과적으로 더 비쌉니다. 상황 맞는 모델·노력 조절이 핵심.

## 체크리스트 — 오늘 당장

- [ ] `MAX_THINKING_TOKENS=10000` 환경변수 설정
- [ ] 프로젝트 `.claude/settings.json`에 기본 모델 Sonnet 고정
- [ ] 자주 쓰는 스킬에 적절한 `model` 프론트매터 추가
- [ ] 오늘 세션 시작·중간·끝 `/cost` 확인 습관
- [ ] 주제 바뀌면 망설이지 말고 `/clear`
- [ ] 긴 작업은 `/compact <무엇을 보존할지>`
- [ ] 탐색 작업은 Explore 서브에이전트에 위임
- [ ] CLAUDE.md 루트 150줄 이하로 슬림
- [ ] 반복되는 결정론 파트 스크립트로 이관

체크리스트 절반만 지켜도 **한 달 뒤 청구서가 체감상 반토막** 나 있을 겁니다.

## 요약

토큰 절약은 **한 번의 큰 최적화**가 아니라 **매일의 작은 습관**입니다. 모델 고르기, `/clear` 누르기, 서브에 위임하기 — 각자는 소소하지만 누적되면 지출 곡선 자체가 바뀝니다.

가장 먼저 하나만 할 수 있다면: **세션 시작할 때 `/cost`, 주제 바뀔 때 `/clear`, 긴 세션 중간에 `/compact`** 이 세 가지. 이것만 해도 체감 30% 이상 절감됩니다.

나머지 최적화는 그 위에 쌓아가세요.

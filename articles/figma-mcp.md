# 피그마 MCP로 디자인과 코드 잇기

> 디자이너의 피그마 파일을 클로드가 직접 읽어 코드로 옮기는 워크플로를 소개합니다.

## 들어가며

프런트엔드 작업에서 가장 시간이 오래 걸리는 일 중 하나는 "디자인을 코드로 옮기는 작업"입니다. 색상 값을 눈대중으로 집어넣고, 여백을 한 픽셀씩 맞추고, 컴포넌트 이름을 수동으로 찾아 쓰는 일은 단순 반복이지만 생각보다 많은 시간을 잡아먹습니다.

피그마에서 공식으로 제공하는 MCP 서버를 연결하면 이 과정이 크게 달라집니다. 클로드가 피그마 파일을 직접 읽어 구조, 토큰, 컴포넌트 매핑까지 파악한 뒤 코드로 변환해 주기 때문입니다. 이 부록에서는 피그마 MCP의 기본 개념과 실전 활용 패턴을 정리합니다.

## 피그마 MCP란

피그마 MCP는 클로드가 피그마 파일에서 다음 정보를 읽어올 수 있게 해주는 공식 서버입니다.

- **디자인 컨텍스트** — 노드의 레이아웃, 스타일, 자식 구조
- **스크린샷** — 해당 노드의 실제 렌더링 이미지
- **메타데이터** — 컴포넌트 이름, 변수(토큰), 제약 조건
- **코드 커넥트** — 피그마 컴포넌트와 코드베이스 컴포넌트의 매핑 정보

이 중 가장 중요한 것은 **코드 커넥트**입니다. 디자인 시스템을 구축한 팀이라면 피그마의 버튼 컴포넌트가 코드베이스의 어떤 React 컴포넌트와 연결되는지 미리 매핑해 둘 수 있고, 클로드는 이 매핑을 따라 곧바로 올바른 컴포넌트 코드를 생성합니다.

## 기본 워크플로

### 1단계: 피그마 URL 준비

피그마에서 변환하고 싶은 프레임이나 컴포넌트를 우클릭한 뒤 "Copy link to selection"을 선택합니다. URL에는 `fileKey`와 `node-id` 두 가지 정보가 담겨 있습니다.

```
https://www.figma.com/design/{fileKey}/{fileName}?node-id={nodeId}
```

### 2단계: 클로드에 URL 전달

클로드 코드에 피그마 URL과 함께 요청을 보냅니다.

```
이 피그마 디자인을 React 컴포넌트로 만들어 줘
https://www.figma.com/design/abc123/Dashboard?node-id=42-137
```

클로드는 `get_design_context` 도구로 해당 노드의 구조, 스타일, 스크린샷을 한 번에 가져옵니다.

### 3단계: 프로젝트에 맞춰 적응

여기가 가장 중요한 단계입니다. 피그마 MCP가 돌려주는 코드는 **참고용**이지 그대로 쓸 최종 결과물이 아닙니다. 클로드는 기본적으로 React + Tailwind 형태로 코드를 생성하지만, 실제 프로젝트에서는 다음과 같이 적응이 필요합니다.

- 이미 존재하는 컴포넌트가 있다면 그것을 재사용
- 프로젝트의 디자인 토큰 시스템에 맞춰 색상/여백 변환
- 팀 컨벤션에 맞는 파일 구조와 네이밍

## 세 가지 응답 유형 구분하기

피그마 MCP가 돌려주는 결과는 프로젝트의 디자인 시스템 성숙도에 따라 크게 세 가지로 나뉩니다. 어떤 유형인지 구분하고 대응해야 좋은 결과가 나옵니다.

### 유형 A: 코드 커넥트가 매핑된 경우

가장 이상적인 경우입니다. 피그마 컴포넌트에 코드베이스 컴포넌트가 매핑되어 있다면 클로드는 그 스니펫을 그대로 활용합니다.

```tsx
import { Button } from '@/components/ui/button';

<Button variant="primary" size="lg">저장하기</Button>
```

팀에서 코드 커넥트를 적극 활용하고 있다면, 클로드가 생성하는 코드의 일관성은 놀라울 정도로 높아집니다.

### 유형 B: 디자인 토큰이 CSS 변수로 노출된 경우

피그마 변수(Variables) 기능을 써서 토큰을 관리한다면, 응답에 `var(--color-primary)` 같은 CSS 변수가 포함됩니다. 이 경우 프로젝트의 토큰 시스템에 매핑만 맞춰주면 됩니다.

### 유형 C: 날 것의 hex 값과 절대 좌표만 오는 경우

디자인이 시스템으로 정리되지 않은 상태입니다. 이럴 때는 MCP가 돌려주는 코드를 그대로 쓰기보다 **스크린샷을 참고하여** 프로젝트의 레이아웃 패턴에 맞게 다시 구성하는 편이 낫습니다.

## 실전 팁

### 팁 1: 한 번에 작은 단위를 요청하라

피그마 전체 페이지를 한 번에 변환하려고 하면 클로드도 사람도 맥락을 놓치기 쉽습니다. 섹션 단위로 나누어 요청하면 품질이 눈에 띄게 좋아집니다.

| 나쁜 예 | 좋은 예 |
|---------|---------|
| "대시보드 전체를 코드로 만들어 줘" | "대시보드 상단의 요약 카드 4개를 컴포넌트로 만들어 줘" |
| "이 피그마 파일을 그대로 구현해 줘" | "로그인 폼 노드만 먼저 변환해 줘" |

### 팁 2: 스크린샷을 함께 확인하라

`get_screenshot` 결과를 같이 보면 레이아웃 의도를 훨씬 정확하게 파악할 수 있습니다. 특히 반응형 동작이나 상태 변화를 추측할 때 도움이 됩니다.

### 팁 3: 코드 커넥트는 팀 단위로 투자할 가치가 있다

개인 작업에서는 코드 커넥트까지 설정하는 게 부담스럽지만, 팀에서 디자인 시스템을 운영한다면 이야기가 다릅니다. 한 번 매핑해 두면 모든 팀원의 피그마→코드 변환 품질이 동시에 올라갑니다. `.figma.ts` 파일 하나로 표현되는 매핑은 관리 비용에 비해 효과가 큽니다.

### 팁 4: 피그마로 되쓰기도 된다

피그마 MCP는 "피그마에서 코드로" 뿐만 아니라 "코드에서 피그마로"도 지원합니다. 이미 구현된 페이지를 피그마로 되돌려 디자이너와 합을 맞추거나, 다이어그램을 FigJam에 자동 생성하는 것도 가능합니다.

## 한계와 주의점

피그마 MCP가 만능은 아닙니다. 다음 상황에서는 기대치를 조정해야 합니다.

- **복잡한 인터랙션** — 애니메이션, 상태 머신, 제스처 같은 동적 요소는 정적 디자인으로만 표현되므로 MCP가 유추할 수 있는 정보에 한계가 있습니다.
- **디자인 시스템이 없는 파일** — 앞서 말한 유형 C의 경우, 자동 변환 품질이 급격히 떨어집니다. 이럴 때는 스크린샷 기반의 수작업이 더 빠를 수 있습니다.
- **접근성과 시맨틱 마크업** — 피그마에는 시맨틱 정보가 거의 없습니다. 생성된 코드에 `<button>`이 아닌 `<div>`가 들어가 있을 수 있으므로 리뷰가 필요합니다.

## 마치며

피그마 MCP는 디자인-개발 협업에서 꽤 오랫동안 해결되지 않던 "번역 비용"을 줄여주는 도구입니다. 완벽한 자동 변환을 기대하기보다, **디자이너의 의도를 빠르게 파악하고 프로젝트의 맥락에 맞게 다시 쓰는 출발점**으로 보는 편이 건강합니다.

책 본문에서 MCP의 개념을 익혔다면, 이 부록을 참고하여 실제 피그마 파일 하나를 골라 변환 실습을 해보세요. 코드 커넥트 설정까지 경험해 보면 팀 단위 디자인 시스템 운영의 감을 잡을 수 있습니다.

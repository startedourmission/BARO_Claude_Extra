# 서브에이전트 설계

> 메인 대화를 깨끗하게 유지하면서 독립 작업을 위임하고, 때로는 **병렬로 돌리는** 구조. 클로드 코드를 고수처럼 쓰는 사람들의 공통 패턴.

## 왜 서브에이전트인가

일을 크게 시키면 문제가 둘 생깁니다.

1. **컨텍스트 오염** — 대량 조사 결과가 히스토리에 쌓여 이후 대화의 품질이 떨어짐
2. **직렬 실행** — 5개 파일을 훑는 동안 다른 일을 못 함

서브에이전트는 둘 다 해결합니다.

- 독립된 컨텍스트에서 조사/작업 → **결과 요약만** 메인으로 돌아옴
- 여러 에이전트를 **동시에** 띄울 수 있음

경험상 "3개 이상의 하위 작업으로 쪼갤 수 있는" 요청은 서브에이전트가 답입니다.

## 빌트인 에이전트

클로드 코드에 기본 내장된 에이전트들:

| 에이전트 | 쓰임새 |
|----------|--------|
| `general-purpose` | 범용. 특별히 지정하지 않으면 기본값 |
| `Explore` | 코드베이스 탐색 전용. Read/Grep/Glob에 최적화 |
| `Plan` | 구현 계획 수립 전용. 코드 쓰지 않고 설계만 |
| `claude-code-guide` | 클로드 코드 자체 사용법 질문 전문 |

호출은 `Agent` 도구로 자연스럽게 이뤄지거나, 대화에서 `/agents`로 목록을 볼 수 있습니다.

## 커스텀 에이전트 만들기

`.claude/agents/` 또는 `~/.claude/agents/`에 마크다운 파일을 두면 됩니다.

`.claude/agents/korean-reviewer.md`:

```markdown
---
name: korean-reviewer
description: 한국어로 코드 리뷰를 수행하는 시니어 엔지니어
tools: Read, Grep, Glob, Bash(git *)
model: sonnet
---

당신은 한국어로 답하는 시니어 백엔드 리뷰어입니다.

리뷰 기준:
1. 버그 가능성 — 엣지 케이스, 동시성, NPE
2. 성능 — 불필요한 N+1, 큰 메모리
3. 가독성 — 네이밍, 함수 크기

답변은 반드시 한국어로, 코드 라인 번호를 꼭 인용.
```

메인 클로드가 "리뷰해 줘"라고 요청받으면 `Agent(subagent_type="korean-reviewer")`로 이 에이전트를 띄웁니다.

### frontmatter 필드

| 필드 | 기능 |
|------|------|
| `name` | 에이전트 이름 |
| `description` | 언제 쓰는지 — 메인이 선택 판단에 사용 |
| `tools` | 이 에이전트가 쓸 수 있는 도구 (공백 또는 콤마) |
| `model` | 전용 모델 (opus/sonnet/haiku) |
| `effort` | 노력 레벨 (low/medium/high/xhigh/max) |
| `allowed-tools` | 묻지 않고 쓸 도구 화이트리스트 |

도구를 좁게 제한할수록 **에이전트의 행동이 예측 가능**해집니다. Explore에게 Write를 주지 않는 게 디폴트인 이유.

## 호출 방법

### 메인에서 직접

메인 클로드가 Agent 도구를 호출할 때:

```
Agent({
  subagent_type: "korean-reviewer",
  description: "auth 모듈 리뷰",
  prompt: "src/auth/ 아래 최근 커밋 변경 사항을 리뷰해 줘. 버그·성능·가독성 순으로."
})
```

에이전트는 **독립 컨텍스트**에서 작업을 수행하고, 최종 결과 메시지 하나를 메인에 돌려줍니다.

### 병렬 실행

독립적인 작업 여러 개는 한 메시지에 여러 Agent 호출을 넣어 **동시에** 띄웁니다.

```
Agent 1: "src/api/ 리뷰"
Agent 2: "src/ui/ 리뷰"
Agent 3: "테스트 커버리지 분석"
```

3배 빠르게 끝나고, 각자 독립 컨텍스트라 서로 간섭 없음.

### 스킬 안에서 `context: fork`

스킬을 서브에이전트에서 실행하려면 프론트매터에 `context: fork`를 추가합니다.

```yaml
---
name: deep-research
description: 주제를 깊게 조사
context: fork
agent: Explore
---

$ARGUMENTS 주제를 깊이 조사하고 요약해 줘.
1. Glob/Grep으로 관련 파일 찾기
2. 각 파일 핵심만 발췌
3. 공통 패턴 도출
```

`/deep-research "인증 플로우"` 실행 시:

1. 새 Explore 에이전트가 포크됨
2. 위 프롬프트를 받아 독립 실행
3. 결과 요약이 메인으로 반환

메인 컨텍스트에 40개 파일이 로드되지 않고 **요약만** 올라옵니다.

### 스킬 프리로드 (반대 방향)

서브에이전트가 특정 스킬을 기본 장착하고 시작하게 할 수도 있습니다.

```markdown
---
name: security-agent
tools: Read, Grep, Bash
skills:
  - owasp-checklist
  - threat-model
---

보안 전문 서브에이전트. 위 스킬들을 컨텍스트에 미리 로드한 채 시작.
```

이 에이전트가 호출되면 `owasp-checklist`와 `threat-model` 스킬 내용이 시스템 프롬프트에 미리 들어갑니다.

## 설계 원칙

### 좁고 명확하게

에이전트마다 **역할 한 문장**으로 요약할 수 있어야 합니다.

- ❌ "general-helper" — 범용이면 그냥 메인이 하면 됨
- ✅ "migration-writer — 스키마 diff 읽고 마이그레이션 스크립트 작성"
- ✅ "changelog-compiler — 커밋 로그 읽고 릴리스 노트 작성"

### 입력·출력 명시

description에 어떤 입력을 받고 어떤 출력을 내는지 분명히 적으세요.

```yaml
description: >
  입력: 작업 브랜치명 (예: feature/login-v2)
  출력: 해당 브랜치의 PR 설명 마크다운.
  사용 시기: PR 생성 직전.
```

메인이 "이 에이전트를 쓸지 말지" 판단할 수 있어야 합니다.

### 도구를 최소로

에이전트에게 필요한 도구만 줍니다. 코드 탐색만 하는 에이전트에게 Write/Edit/Bash를 주면 메인이 해야 할 일까지 대신해서 혼란이 옵니다.

```yaml
# Explore용
tools: Read, Grep, Glob

# 작성용
tools: Read, Write, Edit
```

### 반환 메시지 규약

서브에이전트가 돌려주는 메시지를 **구조화된 형식**으로 받도록 프롬프트에 지시하세요.

```markdown
응답은 반드시 아래 JSON으로:

{
  "summary": "한 문장 결론",
  "findings": [{"file": "...", "line": 12, "issue": "..."}],
  "next_steps": ["..."]
}
```

메인이 파싱해서 후속 행동을 결정하기 쉬워집니다.

## 패턴 1 — Explore → Plan → 실행

대규모 기능 구현의 표준 플로우.

1. **Explore 에이전트** — 관련 파일·기존 패턴 조사 (1개)
2. **Plan 에이전트** — Explore 결과로 구현 계획 수립 (1개)
3. **실행 에이전트들** — Plan의 각 단계를 병렬로 수행 (N개)
4. **메인** — 실행 결과 검토, 최종 조립

이 구조로 가면 메인 컨텍스트는 **계획과 요약**만 담고, 세부는 서브에서 처리.

## 패턴 2 — 병렬 리서치

하나의 질문을 여러 각도에서 동시 조사.

```
Agent 1: "기술 관점으로 — 성능, 확장성, 의존성"
Agent 2: "비즈니스 관점으로 — 리스크, 비용, 타임라인"
Agent 3: "사용자 관점으로 — UX, 학습 곡선, 대체재"
```

세 결과를 메인이 종합. 혼자 순차로 하면 15분 걸릴 일을 5분에.

## 패턴 3 — 검증 루프

작성 에이전트와 리뷰 에이전트를 분리해 돌립니다.

```
Agent A: "인증 모듈 구현"
  → 결과 반환
Agent B (korean-reviewer): "Agent A가 만든 코드 리뷰"
  → 지적 사항 반환
메인: Agent A에게 피드백 전달, 수정 요청
```

자가 검증보다 **분리된 관점**이 버그를 더 잘 잡습니다.

## 패턴 4 — 격리 작업 (worktree)

위험한 실험은 git worktree로 격리된 브랜치에서 돌립니다.

```
Agent({
  subagent_type: "refactor-specialist",
  isolation: "worktree",
  prompt: "src/api/ 전체를 tRPC로 마이그레이션. 실패해도 괜찮음."
})
```

에이전트는 새 worktree에서 작업하고, 성공하면 결과 경로를 반환, 실패/미완료면 자동 정리됩니다.

## 호출 시점 — 언제 서브에이전트, 언제 메인?

| 상황 | 선택 |
|------|------|
| 파일 1-3개 수정 | 메인 |
| 탐색·읽기 위주 (결과 요약만 필요) | Explore 서브 |
| 계획 수립 (빠르고 잘 짜여야) | Plan 서브 |
| 독립적 작업 여러 개 | 병렬 서브 |
| 전문 도메인 리뷰 | 커스텀 서브 |
| 위험한 대규모 리팩터 | worktree 격리 서브 |

## 함정들

### 서브 안에서 또 서브

서브에이전트가 또 다른 서브에이전트를 띄우는 구조는 **원칙적으로 피하세요**. 에이전트 계층이 3단으로 깊어지면 디버깅이 악몽이 됩니다. 필요하다면 메인에서 병렬로 띄우세요.

### description 빈약

`description: "리뷰 에이전트"` 이러면 메인이 언제 써야 할지 몰라서 잘 호출 안 합니다. **트리거 조건**을 명확히.

```yaml
description: >
  코드 리뷰가 필요할 때. 특히 PR 생성 전, 커밋 직전,
  "review this", "리뷰해 줘" 요청이 있을 때.
```

### 과도한 도구 허용

서브에이전트에게 메인과 같은 도구 풀을 주면 경계가 흐려집니다. 서브는 **목적에 필요한 최소 도구**만.

### 큰 결과물 그대로 반환

서브가 "파일 10개 전문"을 결과로 돌려주면 메인 컨텍스트가 오염됩니다. **"요약 + 경로 참조"**로 돌려주도록 프롬프트 지시.

```markdown
결과는 다음 형식으로:
- 핵심 발견 3줄
- 상세는 `~/.claude/logs/explore-$(date).md`에 저장
```

## 요약

서브에이전트는 **역할 분담 엔진**입니다. 메인은 감독, 서브는 실무. 이 계층이 잡히면 클로드 코드는 "혼자 일하는 도구"에서 "작은 팀을 운영하는 도구"로 바뀝니다.

오늘 한 번 해 볼 일: 자주 반복하는 리뷰 프롬프트를 `.claude/agents/reviewer.md`로 추출해 보세요. 다음 PR 만들 때 `Agent` 한 줄로 호출되는 경험이, 서브에이전트의 가치를 즉시 보여줍니다.

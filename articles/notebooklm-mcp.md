# 클로드와 NotebookLM 연결하기

> 할루시네이션 없이 최신 문서를 읽는 법. 클로드 코드가 NotebookLM에 직접 질문을 던지고 답을 받아 코드를 쓰게 만드는 MCP.

## 들어가며

신규 라이브러리나 빠르게 바뀌는 API로 작업해 본 적이 있다면 익숙한 장면일 겁니다. 클로드에게 "n8n으로 워크플로 짜줘"라고 시키면, 실제로는 존재하지 않는 노드 이름이나 함수가 코드에 섞여 있습니다. 할루시네이션이죠. LLM의 학습 시점 이후에 만들어진 API이거나, 내부 전용 문서라서 모델이 애초에 모르는 경우입니다.

[PleasePrompto/notebooklm-mcp](https://github.com/PleasePrompto/notebooklm-mcp)는 이 문제를 **"AI가 AI에게 묻게 하자"**는 방식으로 푼 MCP 서버입니다. 클로드 코드가 직접 구글의 [NotebookLM](https://notebooklm.google.com/)에 질문을 던지고, Gemini가 내가 올려 둔 문서만 근거로 답을 합성해 돌려줍니다. 할루시네이션이 구조적으로 차단된 답이 클로드에게 전달되고, 클로드는 그 답을 근거로 코드를 씁니다.

이 글에서는 이 MCP가 왜 필요한지, 어떻게 설치하고 쓰는지, 기존 접근법과 무엇이 다른지를 정리합니다.

## 왜 이게 필요한가

클로드 코드에게 "이 로컬 문서 폴더를 참고해서 짜줘"라고 시킬 때 실제로 일어나는 일:

- **토큰 폭발** — 여러 파일을 반복해서 읽느라 컨텍스트가 거대해짐
- **연결 누락** — 키워드 기반 검색이라, 문서 A의 개념과 문서 B의 개념이 같은 것을 가리킨다는 걸 놓침
- **빈 곳 채우기** — 못 찾으면 그럴듯하게 지어냄 (가장 치명적)
- **반복 비용** — 질문마다 같은 파일을 다시 읽음

로컬 RAG를 구축해도 문제가 완전히 사라지진 않습니다. 임베딩·청크 전략·벡터 DB 운영의 복잡성이 새로 생기고, 검색 누락은 여전히 남습니다.

## NotebookLM이라는 해결책

NotebookLM은 구글이 Gemini 2.5 위에 만든 **제로 할루시네이션 지식 베이스**입니다. 특징은 단순합니다.

- 업로드한 문서에서만 답변. 모르는 건 "모른다"라고 답함
- 최대 50개 이상의 문서를 교차 참조
- 모든 답변에 citation(근거 표시) 포함
- 벡터 DB, 임베딩, 청크 설정 전혀 필요 없음

여기까지는 NotebookLM 자체의 장점입니다. 문제는 **NotebookLM은 웹 UI**라는 점이죠. 질문할 때마다 에디터 ↔ 브라우저 복사-붙여넣기를 반복해야 합니다. 이 MCP는 그 수작업을 없앱니다.

```
내 작업 → 클로드 → MCP → Chrome 자동화 → NotebookLM → Gemini → 내 문서
                                                     ↓
내 작업 ← 정확한 코드 ← 클로드 ← MCP ← Chrome ← NotebookLM ← Gemini
```

클로드는 필요하면 **여러 번 후속 질문**을 던지고, 그때마다 Gemini가 내 문서에 근거해 답을 줍니다. 최종적으로 클로드가 가진 지식은 "내 문서를 깊이 이해한 상태"가 되고, 그 위에서 코드를 씁니다.

## 설치

### 클로드 코드

```bash
claude mcp add notebooklm npx notebooklm-mcp@latest
```

### Codex

```bash
codex mcp add notebooklm -- npx notebooklm-mcp@latest
```

### Cursor

`~/.cursor/mcp.json`에 추가:

```json
{
  "mcpServers": {
    "notebooklm": {
      "command": "npx",
      "args": ["-y", "notebooklm-mcp@latest"]
    }
  }
}
```

다른 MCP 클라이언트(VS Code, amp, Gemini 등)도 지원합니다. 설치가 끝나면 대화창에서 바로 쓸 수 있습니다.

## 초기 세팅 4단계

### 1단계: 로그인

```text
NotebookLM에 로그인해 줘
```

Chrome 창이 뜨고 구글 계정으로 로그인하면 끝입니다. 한 번 인증하면 세션이 유지됩니다.

> ⚠️ **전용 계정 권장**: 브라우저 자동화이므로 구글이 주 계정을 플래그할 가능성이 있습니다. 자동화 전용 구글 계정을 만들어 쓰는 것이 안전합니다.

### 2단계: NotebookLM에 지식 베이스 만들기

[notebooklm.google.com](https://notebooklm.google.com/)에서 새 노트북을 만들고 자료를 업로드합니다.

- PDF, Google Docs, 마크다운
- 웹사이트, 깃허브 저장소
- 유튜브 영상
- 한 노트북에 여러 소스를 섞어도 됨

공유 버튼 → "링크가 있는 모든 사용자" → 링크 복사.

### 3단계: 라이브러리에 등록

클로드에게 링크를 주고 태그와 함께 등록하게 시킵니다.

```text
이 링크를 라이브러리에 "react, frontend, components" 태그로 추가해 줘
https://notebooklm.google.com/notebook/abc123...
```

노트북을 여러 개 저장해 두면, 이후 작업에 맞는 노트북을 클로드가 **자동 선택**합니다.

### 4단계: 질문하기

```text
[라이브러리] 리액트 노트북을 참고해서
드래그 앤 드롭 가능한 칸반 보드 컴포넌트를 만들어 줘.
먼저 노트북에 질문 몇 개 해서 API를 정확히 확인한 뒤 코드 작성.
```

클로드가 NotebookLM에 다단계로 질문하고, 답을 받아가며 코드를 완성합니다.

## 실전 예: n8n 워크플로

README에 소개된 예시를 거의 그대로 옮깁니다. n8n API는 새로워서 클로드가 노드 이름을 자주 지어냅니다.

1. n8n 공식 문서를 통째로 덤프해 하나의 NotebookLM 노트북에 업로드
2. 클로드에게 지시: "이 노트북 참고해서 Gmail 스팸 필터 워크플로를 만들어 줘"

클로드와 NotebookLM 사이의 대화는 이렇게 진행됩니다.

```
클로드 → "n8n에서 Gmail 통합은 어떻게 하나요?"
NotebookLM → "Gmail Trigger 노드의 폴링 모드 또는 Gmail 노드의 Get Many를..."

클로드 → "base64 인코딩된 이메일 본문은 어떻게 디코딩?"
NotebookLM → "payload.parts의 body는 base64url입니다. Function 노드로..."

클로드 → "OpenAI 응답을 JSON으로 파싱하려면?"
NotebookLM → "responseFormat을 json으로 설정하고, IF 노드에서 {{ $json.spam }}..."

클로드 → "API 실패 시 에러 처리는?"
NotebookLM → "Error Trigger 노드와 Continue On Fail을 함께..."

클로드 → ✅ "완성된 워크플로 JSON입니다..."
```

첫 시도에 작동하는 워크플로가 나옵니다. 지어낸 API를 디버깅하는 시간이 0입니다.

## 기존 방법과 비교

| 방식 | 토큰 비용 | 세팅 시간 | 할루시네이션 | 답변 품질 |
|------|-----------|-----------|--------------|-----------|
| 문서를 클로드에 직접 주기 | 매우 높음 (반복 읽기) | 즉시 | 있음 (빈 곳 채움) | 검색에 좌우 |
| 웹 검색 | 중간 | 즉시 | 높음 (출처 불안정) | 들쭉날쭉 |
| 로컬 RAG | 중~고 | 수 시간 (임베딩·청크) | 중간 (검색 누락) | 세팅에 좌우 |
| **NotebookLM MCP** | **최소** | **5분** | **없음 (모르면 모른다고)** | **전문가 수준 합성** |

세 가지 결정적 차이가 있습니다.

1. **사전 처리** — Gemini가 업로드 시점에 문서를 이해해 둠. 질의 시점에 다시 처리할 필요가 없음
2. **교차 참조** — 50개 넘는 문서 간 연결을 자동으로 맺음
3. **인용 포함** — 답변마다 어느 소스의 어느 부분인지 표기

## 라이브러리 관리

한 번 저장한 노트북 링크는 라이브러리에 들어가 여러 도구(클로드 코드, Codex, Cursor)에서 **공유**됩니다.

```text
현재 등록된 노트북 목록 보여줘

React 노트북의 태그를 업데이트해 줘:
"react, frontend, hooks, v19"

더 이상 안 쓰는 Vue 노트북은 지워줘
```

작업 맥락("지금 리액트 컴포넌트 만드는 중")에 맞춰 클로드가 **관련 노트북을 자동 선택**합니다. 여러 프로젝트를 오갈 때 특히 편합니다.

## 도구 프로파일로 토큰 절약하기

MCP 서버는 전체 16개 도구를 노출하지만, 작업 유형에 따라 일부만 로드해 컨텍스트를 절약할 수 있습니다.

| 프로파일 | 도구 수 | 언제 씀 |
|----------|---------|---------|
| **minimal** | 5 | 질의 전용 — 등록된 노트북에 질문만 |
| **standard** | 10 | 라이브러리 관리 포함 |
| **full** | 16 | 관리·청소·세션 리셋까지 모두 |

설정은 CLI로 바꿀 수 있습니다.

```bash
npx notebooklm-mcp config set profile minimal
npx notebooklm-mcp config set profile standard

# 특정 도구만 끄기
npx notebooklm-mcp config set disabled-tools "cleanup_data,re_auth"
```

환경 변수 `NOTEBOOKLM_PROFILE`, `NOTEBOOKLM_DISABLED_TOOLS`로 오버라이드할 수도 있습니다.

## 한계와 주의점

### 브라우저 자동화의 운명

이 MCP는 Chrome 자동화로 NotebookLM UI를 조작합니다. 저자가 타이핑 속도·마우스 움직임·지연을 "인간처럼" 흉내 내는 휴머나이제이션을 넣어두긴 했지만, **구글이 자동화를 탐지할 가능성은 0이 아닙니다.** 주 계정 대신 자동화 전용 계정을 쓰는 것이 권장됩니다.

### 무료 티어 한도

NotebookLM 무료 티어에는 구글 계정당 일일 질의 한도가 있습니다. 장시간 리서치에는 계정 전환 기능이 유용합니다.

### 제로 할루시네이션 ≠ 만능

NotebookLM은 **모르면 모른다고 답합니다.** 덕분에 할루시네이션은 없지만, 내 문서에 답이 없는 질문에는 유용한 답을 받을 수 없습니다. 좋은 답의 전제는 **좋은 소스 큐레이션**입니다.

### 보안

Chrome은 로컬에서 돕니다. 자격 증명이 저자 서버로 나가지 않습니다. 다만 노트북 공유 링크를 라이브러리에 넣어두는 구조이므로, 기밀 문서는 공유 링크 자체를 가볍게 취급하지 마세요.

## 카파시 LLM 위키와 결합하기

같은 부록의 ["카파시의 LLM 위키 만들기"](./karpathy-llm-wiki)와 이 MCP는 상호보완적입니다.

- **카파시 위키** — 지식을 **마크다운 파일로 내 저장소 안**에 축적. 에이전트가 직접 읽고 수정
- **NotebookLM MCP** — 지식을 **Gemini가 이해한 채로 클라우드에** 두고, 필요할 때 대화로 꺼냄

로컬 저장소에 둘 자료는 카파시 위키로, 크고 자주 조회하는 외부 문서는 NotebookLM으로 나눠 두면 좋습니다. 에이전트는 두 가지를 번갈아 참조하며 작업합니다.

## 클로드 코드 스킬 버전

같은 저자가 [notebooklm-skill](https://github.com/PleasePrompto/notebooklm-skill)이라는 클로드 코드 전용 스킬도 배포했습니다. MCP 버전과 비교:

- **MCP 버전 (이 글의 대상)** — 지속 세션, 클로드 코드/Codex/Cursor 등 광범위 지원
- **스킬 버전** — 파이썬 기반, 세팅이 더 단순, 클로드 코드 전용

여러 에이전트를 오가며 쓰는 팀이라면 MCP, 클로드 코드만 쓰는 개인이라면 스킬 쪽이 가볍습니다.

## 마치며

"LLM은 모르면 지어낸다"는 본질적 한계를 가장 현실적으로 푸는 방법은 **LLM을 믿지 않는 설계**입니다. NotebookLM MCP가 그 설계를 구체화합니다. 클로드가 혼자 판단하지 않고, 매번 **"이 문서에 뭐라고 쓰여 있어?"**를 NotebookLM에게 확인한 뒤 코드를 씁니다.

새 라이브러리를 배우거나 내부 API로 작업할 때 한 번 세팅해 두면, "지어낸 API를 디버깅하는 시간"이 통째로 사라집니다. 그게 이 도구의 본질적 가치입니다.

```bash
# 30초면 끝
claude mcp add notebooklm npx notebooklm-mcp@latest
```

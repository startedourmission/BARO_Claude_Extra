# 모델·사고 모드·비용

> Opus는 언제, Sonnet은 언제, Haiku는 언제. `ultrathink`는 얼마나 비싼가. 토큰을 아끼는 프롬프트 캐싱은 어떻게.

## 모델 계보

Claude 4 계열(2026년 초 기준) 대표:

| 모델 | 위치 | 특징 |
|------|------|------|
| **Opus 4.7** | 최강 | 복잡 추론, 장문 코드, 긴 컨텍스트 |
| **Sonnet 4.6** | 균형 | 대부분 작업의 스윗스팟. 속도·품질·가격의 밸런스 |
| **Haiku 4.5** | 경량 | 빠름·저렴. 단순 반복, 분류, 요약 |

각 모델은 "4.x" 숫자에 맞춰 세대 업그레이드. 시간 지나면 4.8·4.9 등이 나옵니다. 현재 설치본의 최신 모델은 `/model`로 확인 가능.

### 1M 컨텍스트 변형

일부 Opus·Sonnet은 "1M 컨텍스트" 변형이 있습니다. `claude-opus-4-7[1m]` 같은 식으로 접미사. 수십만 토큰의 입력을 한 번에 처리할 때 필요.

## `/model` 커맨드

세션 도중 모델을 바꿉니다.

```
/model
```

메뉴에서 고르거나:

```
/model opus
/model sonnet
/model haiku
```

### 작업 성격별 권장

| 작업 | 추천 모델 |
|------|-----------|
| 아키텍처 설계, 대규모 리팩터 | Opus |
| 일상 코딩, 리뷰, 디버깅 | Sonnet |
| 이름 짓기, 한 줄 요약, 분류 | Haiku |
| 긴 문서 요약 (50p+) | Sonnet 1M |
| 법률/재무 정밀 추론 | Opus |
| CI 자동 태깅/라벨링 | Haiku |

팀 세션에서는 **Sonnet 기본 + 필요할 때만 Opus로 격상**이 비용·품질 균형이 좋습니다.

### CLI에서 지정

```bash
claude -p "간단 작업" --model haiku
claude --model opus
```

`ANTHROPIC_MODEL` 환경변수로 기본값을 바꿀 수도 있습니다.

## 사고 모드 (Extended Thinking)

프롬프트에 특정 키워드를 포함하면 모델이 **내부 사고**를 길게 수행합니다. 더 좋은 답을 내는 대신 토큰 소비가 늡니다.

| 키워드 | 상대적 사고량 |
|--------|---------------|
| (없음) | 기본 |
| `think` | 기본의 ~4배 |
| `think hard` | 기본의 ~10배 |
| `think harder` | 기본의 ~15배 |
| `ultrathink` | 최대 |

### 쓰기 예

```
이 버그의 근본 원인을 찾아 줘. think hard.

이 아키텍처 결정의 장단점을 전부 따져 줘. ultrathink.
```

### 언제 쓸까

**쓸 만한 상황:**

- 복잡한 디버깅 (동시성, 메모리, 플레이키 테스트)
- 대규모 리팩터링 전략
- 여러 제약이 얽힌 설계 결정
- 성능 최적화 분석

**쓰지 말 것:**

- 단순 코드 생성 (그냥 쓰면 됨)
- 번역·요약 (사고가 크게 도움 안 됨)
- 긴 CI 루프 (비용 폭발)

### Interleaved Thinking

최신 모델(Opus 4.7, Sonnet 4.6)부터는 **도구 호출 사이사이**에도 사고가 들어갈 수 있습니다. 별도 설정 없이 자동 활성. 예: 파일을 읽은 **후** "이 구조를 어떻게 바꿀까" 생각을 한 다음 Edit을 호출.

이 덕에 에이전트적 작업의 품질이 크게 올랐습니다. 단, 비용도 늘어나므로 `MAX_THINKING_TOKENS` 환경변수로 상한을 둘 수 있습니다.

## 비용 추적 — `/cost`

현재 세션의 토큰·비용을 확인.

```
/cost
```

표시되는 정보:

- 입력·출력 토큰 누적
- 프롬프트 캐시 히트율
- 추정 비용 (USD)
- 세션 시작 이후 경과 시간

## 프롬프트 캐싱

**긴 시스템 프롬프트·CLAUDE.md·문서를 5분간 캐시**해 재사용 시 입력 토큰 비용을 **90% 절감**합니다.

### 자동 적용

클로드 코드는 CLAUDE.md·스킬 본문·대화 히스토리를 알아서 캐시에 올립니다. 개발자가 API를 직접 부를 때처럼 cache breakpoint를 고민할 필요가 없습니다.

### 캐시 히트율 확인

```
/cost
```

히트율이 낮다면:

1. 5분 이상 쉬었다 복귀한 것 — 캐시 TTL 초과
2. CLAUDE.md가 자주 바뀜 — 캐시 무효화 자주 발생
3. 세션마다 `/clear` — 매번 새 캐시

### 최적화 팁

- **`/clear`를 신중히** — 이후 첫 요청은 캐시 미스
- **CLAUDE.md를 안정적으로** — 수시로 바꾸지 말고 주기적으로 정리
- **긴 작업은 연속으로** — 쉬지 않고 이어가면 캐시가 계속 hit

## 비용 관리 실전

### 예산 모니터링

- 매일 `/cost` 한두 번 습관화
- 세션 시작 전 모델 확인 (실수로 Opus로 Haiku 작업 돌리면 비쌈)
- 큰 리포트 생성은 파일로 (대화 반복 피함)

### 모델 자동 라우팅 패턴

자주 쓰는 패턴: 스킬별로 적절한 모델 고정.

```yaml
---
name: quick-rename
description: 변수명 리네이밍
model: haiku
---
```

```yaml
---
name: architecture-review
description: 큰 그림 설계 리뷰
model: opus
effort: high
---
```

매번 `/model` 치지 않아도 맥락에 맞게 자동 전환.

### 무제한 플랜 vs API 종량제

Claude Code는 두 가지 결제 방식 지원:

- **Max 플랜** — 월 고정 금액, 모델 제한 없음
- **API 종량제** — 키로 인증, 토큰당 과금

**Max 플랜이 유리한 경우:**

- 매일 몇 시간씩 쓰는 개발자
- Opus를 자주 쓰는 팀

**API 종량제가 유리한 경우:**

- 가끔만 쓰는 사용자
- Haiku 위주 자동화
- CI에서 간헐 호출

### 엔터프라이즈: Bedrock·Vertex

조직 보안·감사 요구로 AWS Bedrock이나 Google Vertex 경유가 필요하면:

```bash
export CLAUDE_CODE_USE_BEDROCK=1
# 또는
export CLAUDE_CODE_USE_VERTEX=1
```

비용은 각 클라우드의 청구서로 들어가지만 데이터 거버넌스 요건을 맞추는 핵심 경로입니다.

## 모델별 프롬프팅 팁

### Opus용

- 길고 복잡한 지시 가능 — 전부 따라감
- "ultrathink"와 조합하면 설계 품질 향상
- 대신 속도가 느림 (응답 수십 초 걸릴 수 있음)

### Sonnet용

- 명확한 단계 제시가 좋음
- 애매한 요청에 창의적 답을 잘 냄
- 대부분 경우의 기본값

### Haiku용

- 지시는 **짧고 명확**하게
- 긴 문서·복잡 추론은 못 따라감
- 속도가 매우 빠름 (대화형 느낌 강함)

## 함정들

### "최강 모델 = 언제나 최선" 오해

Haiku가 더 빠르고 저렴해서 작업 성격에 따라 **Haiku가 더 잘 맞는** 경우가 많습니다. 단순 분류·요약은 Haiku가 훨씬 가성비.

### 사고 모드 남발

모든 프롬프트에 `ultrathink`를 붙이면 비용이 폭발합니다. **판단이 필요한 지점에만** 넣으세요.

### 캐시 맹신

5분 TTL이라 점심 먹고 와서는 캐시가 날아가 있습니다. 첫 복귀 요청의 비용은 높을 수 있음.

### 1M 모델 기본값

1M 변형은 비쌉니다. 꼭 필요한 세션에만 쓰고, 일상에서는 기본 컨텍스트.

### 모델 스위칭 망각

Opus로 큰 설계 했다가 그대로 두고 간단한 이름 짓기에 계속 Opus를 쓰고 있는 경우가 흔합니다. 작업 끝에 `/model`로 재설정 습관.

## 요약

모델·사고·비용은 **세 개의 다이얼**입니다. 한 번 세팅해 두고 끝이 아니라, 작업 성격마다 돌리는 게 정답.

권장 디폴트:

- 일상: **Sonnet + 기본 사고**
- 복잡: **Sonnet + think hard** 또는 **Opus + 기본**
- 최고 난이도: **Opus + ultrathink**
- 가벼운 자동화: **Haiku + 기본**

매일 `/cost` 확인, 주마다 패턴 돌아보기, 월마다 청구서 대비 실제 가치 평가. 세 주기만 지키면 비용 관리가 어렵지 않습니다.

# CLAUDE.md와 메모리 시스템

> 세션마다 "그거 매번 말 안 해도 알아들었으면" 싶은 맥락을, 한 번 적어 두면 클로드가 항상 읽는 구조. 클로드 코드의 가장 기본이자 가장 중요한 설정.

## 왜 필요한가

코딩 세션을 열면 처음엔 "이 프로젝트는 Next.js 14", "커밋은 한국어", "테스트는 bun test" 같은 설명을 반복합니다. 사람이 바뀌지 않는 규칙을 매번 타이핑하는 건 낭비입니다.

CLAUDE.md는 **세션 시작 시 자동으로 로드되는 지침 파일**입니다. 여러 계층에서 읽혀 서로 병합되고, 클로드는 어떤 요청을 받든 이 파일들의 규칙을 기본 전제로 깔고 답합니다.

## 계층 구조

클로드 코드는 세션이 시작될 때 아래 순서로 메모리를 수집합니다.

| 계층 | 경로 | 적용 범위 |
|------|------|-----------|
| 엔터프라이즈 | OS별 관리 경로 | 조직 전체 (읽기 전용) |
| 개인 | `~/.claude/CLAUDE.md` | 내 모든 프로젝트 |
| 프로젝트 | `<repo>/CLAUDE.md` | 이 프로젝트 (git 커밋 대상) |
| 하위 디렉터리 | `<repo>/packages/foo/CLAUDE.md` | 이 폴더의 파일을 다룰 때 |

**충돌 시 더 구체적인(아래) 계층이 이깁니다.** 예: 개인 CLAUDE.md에 "커밋은 영어"라고 했지만 프로젝트 CLAUDE.md에 "커밋은 한국어"라고 썼으면, 그 프로젝트에서는 한국어가 이깁니다.

## 자동 로드 범위

프로젝트 CLAUDE.md는 **세션을 열 때 현재 디렉터리의 CLAUDE.md + 상위 디렉터리까지 추적**해 올립니다. 즉 `packages/frontend/`에서 클로드를 열면:

- `packages/frontend/CLAUDE.md`
- `packages/CLAUDE.md`
- `<repo-root>/CLAUDE.md`

세 개가 다 로드됩니다 (있는 것만). 모노레포에서 **공통 규칙은 루트에, 특이 규칙은 패키지에** 두는 분리가 자연스럽습니다.

### --add-dir 디렉터리의 CLAUDE.md

`claude --add-dir ~/other-repo`로 외부 경로를 연결하면, 그 경로의 `.claude/skills/`는 자동 로드되지만 **`CLAUDE.md`는 기본적으로 로드되지 않습니다.** 필요하면 환경 변수로 켭니다:

```bash
export CLAUDE_CODE_ADDITIONAL_DIRECTORIES_CLAUDE_MD=1
```

## 쓰는 법

### 기본 템플릿

```markdown
# 프로젝트 개요

Next.js 14 App Router 기반 대시보드. 주요 경로:
- `app/` — 라우트와 페이지
- `components/ui/` — shadcn. 직접 수정 금지
- `lib/api/` — 백엔드 클라이언트

# 스택

- Node 20, pnpm 9
- TypeScript 5.4 strict
- Tailwind v4
- 테스트: Vitest

# 규칙

- 커밋 메시지는 한국어
- `any` 타입 금지
- 새 의존성 추가 시 먼저 물어보고 설치
- UI 문자열은 `messages/*.json`에만

# 명령

- 개발: `pnpm dev`
- 테스트: `pnpm test`
- 린트: `pnpm lint`
- 빌드: `pnpm build`
```

규칙은 **짧은 문장 단위**로 쓰세요. 긴 문단은 LLM이 일부만 반영합니다.

### @import 문법

다른 문서를 CLAUDE.md에서 끌어올 수 있습니다.

```markdown
# 참고

@docs/architecture.md
@.cursorrules
```

아키텍처 문서나 기존 `.cursorrules`를 그대로 재활용할 때 편합니다. 여러 도구(Cursor, Windsurf, Codex)를 혼용하는 팀이라면 한 장의 "마스터 규칙 파일"을 두고 각자 도구가 @import하게 만드는 전략이 깔끔합니다.

### # 문법 — 대화 중에 바로 저장

입력창에 `#`을 붙이고 문장을 쓰면, 그 내용을 어느 CLAUDE.md에 저장할지 묻는 메뉴가 뜹니다.

```
# 이 프로젝트에선 date-fns 대신 dayjs를 쓴다
```

선택지: 개인 / 프로젝트 / 하위 디렉터리. 대화 중 발견한 규칙을 즉시 박제할 때 유용합니다.

### /memory 커맨드

`/memory`를 치면 현재 적용 중인 CLAUDE.md 파일들을 에디터로 바로 엽니다. 계층 구조가 한눈에 보여 편집 대상을 빠르게 고를 수 있습니다.

### /init 커맨드

빈 프로젝트에서 `/init`을 치면 클로드가 프로젝트 구조를 훑어본 뒤 CLAUDE.md 초안을 써 줍니다. 처음 시작할 때 좋은 출발점.

## AGENTS.md 표준

2025년부터 여러 도구(Claude Code, Cursor, Codex, Gemini CLI 등)가 `AGENTS.md`라는 **공통 표준**을 도입하기 시작했습니다. 역할 분담:

| 파일 | 누가 읽나 | 무엇을 |
|------|-----------|--------|
| `AGENTS.md` | 모든 코딩 에이전트 | 프로젝트를 **어떻게 만들지** |
| `DESIGN.md` | 디자인 에이전트 | 프로젝트가 **어떻게 보여야 하는지** |
| `CLAUDE.md` | 클로드 코드 | 위 둘을 포함한 통합 지침 |

클로드 코드는 `CLAUDE.md`를 우선 읽지만, 공용 프로젝트라면 **`AGENTS.md`에 공통 규칙**을 두고 **`CLAUDE.md`에서 그걸 @import**하는 구성이 팀원의 도구 다양성과 호환됩니다.

```markdown
# CLAUDE.md

@AGENTS.md

## 클로드 전용 규칙

- /ko-review 커맨드로 먼저 점검
- 서브에이전트는 Explore만 자동 호출 허용
```

## 실전 작성 팁

### 팩트 vs 절차

- **팩트** — 프로젝트 스택, 디렉터리 규칙, 네이밍 컨벤션. CLAUDE.md에 그대로 적음
- **절차** — "배포할 때 이 순서로" 같은 단계형. 이건 **스킬(`/deploy` 같은)**로 빼는 게 낫습니다

CLAUDE.md에 "배포 10단계"를 풀어 쓰면 모든 세션의 컨텍스트가 무거워집니다. 절차성은 슬래시 커맨드로.

### 경로 스코프 활용

`<repo>/packages/frontend/CLAUDE.md`:

```markdown
# 프런트엔드 전용 규칙

- 상태는 @observable 기반
- 서버 컴포넌트 기본, 'use client'는 불가피한 경우만
- 스타일은 Tailwind v4 arbitrary value 금지
```

루트 CLAUDE.md에 다 쓰지 말고 필요한 폴더에 두세요. 백엔드 파일 다룰 때 프런트 규칙이 컨텍스트에 안 올라와 토큰을 아낍니다.

### 마크다운 헤더 제대로 쓰기

클로드는 마크다운 구조를 보고 중요도·그룹핑을 인식합니다. `## 금지`, `## 권장` 같은 명확한 헤더를 쓰면 더 잘 따릅니다.

```markdown
## ❌ 금지

- `any` 타입
- `console.log` 남기기

## ✅ 권장

- 옵셔널 체이닝 (`?.`)
- 얼리 리턴
```

### 길이 관리

CLAUDE.md 전체가 너무 길면 클로드가 앞부분에만 주목합니다. 목표: **루트 CLAUDE.md 150줄 이내**. 초과하면 하위 폴더·AGENTS.md·별도 문서로 분리.

## 안티패턴

### 사고력 망치는 지침

```markdown
# 금지
- 코드를 절대 수정하지 마라
- 질문에만 답하라
```

이렇게 써 두면 정작 구현이 필요한 순간에도 작업을 거부합니다. **상황별 규칙**("실험 폴더에서는 `// PROTOTYPE: ok` 주석이 있으면 임시 코드 허용")으로 쓰세요.

### 이전 대화 요약 박제

"어제 이런 결정을 했다"는 내용은 CLAUDE.md에 넣지 마세요. **메모리는 영속 규칙 전용**. 결정 이력은 커밋 메시지·ADR(아키텍처 결정 기록)·이슈 트래커로.

### 거대한 단일 파일

모든 규칙을 루트 한 파일에 몰아넣으면 관리가 무너집니다. 도메인별로 쪼개서 @import.

```
CLAUDE.md              # 7줄, @import만
docs/stack.md          # 스택 정보
docs/conventions.md    # 코드 컨벤션
docs/git.md            # 커밋·PR 규칙
```

### 혼자만 아는 약어

```markdown
- PRD 쓸 때는 RICE 프레임워크
```

줄임말은 설명을 같이 쓰세요. LLM이 RICE가 뭔지 모를 수 있습니다. 환각을 유발합니다.

## 메모리 업데이트 전략

CLAUDE.md는 **살아 있는 문서**여야 합니다. 권장 리듬:

- **매 기능 끝날 때** — 새로 정한 컨벤션을 `#` 문법으로 즉시 기록
- **스프린트 회고** — 파일을 훑어 중복·모순·오래된 항목 정리
- **분기마다** — 근본 원칙은 유지하되, 표현·분량은 리라이트

## 요약

| 상황 | 어디에 |
|------|--------|
| 이 프로젝트만의 규칙 | 프로젝트 CLAUDE.md |
| 이 프로젝트의 특정 모듈 규칙 | 해당 폴더 CLAUDE.md |
| 내 개인 스타일·선호 | `~/.claude/CLAUDE.md` |
| 단계별 절차 | 슬래시 커맨드·스킬 |
| 일회성 맥락 | 대화 안에서만 |
| 공용 도구 표준 | AGENTS.md → CLAUDE.md에서 @import |

메모리 시스템은 "**LLM에게 우리 집 규칙을 알려 주는 냉장고 자석**"입니다. 한 번 붙여 두면 세션마다 설명 안 해도 됩니다. 프로젝트 시작할 때 10분만 투자해 CLAUDE.md를 만들어 두면, 이후 한 달 동안 매일 10분씩 아낍니다.
